# Bug Fixes — 2026-05-27

Four reported interaction / scroll issues plus a targeted audit pass of
the surrounding flows. All changes are **additive**: one `<style>` block
and one `<script>` block, both placed at the very end of
`public/index.html` directly above the `<footer>`. Deleting that block
fully reverts to the previous behavior.

The patch is marked in the source with
`<!-- BUGFIX PATCH — 2026-05-27 -->` so it is easy to locate.

---

## Issue #1 — All Questions: image click opens the Answered page

### Reported behavior

In **All Questions** (the read-only page for everyone), clicking the
image thumbnail, the rendered image link, or the "▶ 📷 …" expand
chevron was incorrectly opening the Answered page (a navigation), not
just previewing the image.

### Root cause

Each row in the All Questions table is rendered with an inline
`onclick` on the `<tr>` itself:

```html
<tr onclick="allQRowClick(this.dataset.taskId, this.dataset.status)">
  …
  <td class="td-q">…escaped question text + post-rendered image attachments…</td>
  …
</tr>
```

`allQRowClick` routes Answered rows to `showPage('faq')`.

Image attachments are inserted into `.td-q` after the row is rendered
by the "[Attachment: …](url)" → `<img>` post-processor introduced in
the 2026-05-26 fix, and then wrapped in a `<details class="img-collapse">`
by the image-expand script lower in the file. None of these inner
elements called `stopPropagation()` on click, so every click on the
image, on the expand chevron, or on the captioned image bubbled up to
the `<tr>` and fired the row navigation.

The global `document.addEventListener('click', …)` that opens images in
a new tab also fired, but only *after* the row's onclick had already
swapped pages — so what the user saw was "I clicked the image and it
navigated me away."

### Fix

A bubble-phase click listener is attached directly to each
`<details class="img-collapse">` wrapper and to each
`<img class="bugfix-attachment-image">` as the image-collapse script
materializes them. The listener:

1. (on `<img>`) calls `window.open(src, '_blank', 'noopener,noreferrer')`,
2. calls `e.stopPropagation()` so the click never reaches the `<tr>`.

The native `<summary>` toggle behavior on `<details>` is preserved —
`stopPropagation()` does not cancel the default action.

> Why bubble-phase, not capture-phase: the 2026-05-26 bug fix removed a
> document-level *capture-phase* listener that was calling
> `stopPropagation()` and accidentally killing every inline `onclick`
> inside comment forms. We do not want to reintroduce that class of
> bug, so the new patch is scoped to specific elements only and uses
> the bubble phase, where the event reaches the target's own handlers
> first.

## Issue #2 — Review page: clicking image opens BOTH the reviewer modal AND toggles preview; collapsing reopens the modal

### Reported behavior

In the **Reviewer Dashboard**:

- Clicking the image thumbnail or the "▶ 📷 …" chevron opened the
  Reviewer Answer Form modal *and* expanded/collapsed the image.
- Clicking the chevron a second time to collapse the image *also*
  reopened the Reviewer modal.

### Root cause

Identical to #1 — the Review page renders rows with
`<tr onclick="openModal('${q.id}')">`. Every click inside the row
(including the `<summary>` expand chevron, including the `<img>` and
its caption) bubbled to the row and called `openModal()`.

Because `<summary>` toggles its `<details>` parent on every click, this
manifested as "image expanded *and* modal opened" on the way down, and
"image collapsed *and* modal opened" on the way back up.

### Fix

Same patch as #1. The `<details class="img-collapse">` wrapper now
stops the click before it reaches the `<tr>`. The summary's default
toggle still fires; the modal does not.

## Issue #3 — Team Chat should scroll like Slack

### Reported behavior

The whole **Team Chat** page scrolled as one block. There was no
distinction between the channel sidebar, the message list, the chat
header, and the composer — long conversations pushed the composer off
the bottom of the viewport, and scrolling the channel list also
scrolled the messages.

### Root cause

The base CSS used **`min-height`** instead of **`height`** on the
chat shell and main pane:

```css
.collab-shell { …  min-height: calc(100vh - 185px); }
.collab-main  { …  min-height: 680px; }
.collab-channel-list { …  max-height: 56vh; overflow: auto; }
```

With `min-height`, the shell *expands* to fit its contents. Adding
messages made the whole page grow, which scrolled the body — and
because the page was now taller than the viewport, the sticky-ish
header had nothing to stick to relative to.

### Fix

`#page-collab.active` is turned into a flex column container that is
**exactly** `100vh − nav` tall (using the existing `--nav-height`
custom property = `60px`). Inside it:

| Pane                  | Sizing                                      |
|-----------------------|---------------------------------------------|
| `.collab-hero`        | `flex: 0 0 auto` (natural height)           |
| `.collab-shell`       | `flex: 1 1 auto`, `overflow: hidden`        |
| `.collab-sidebar`     | column-flex, `overflow: hidden`             |
| `.collab-sidebar-head`| `flex: 0 0 auto` (sticky-feeling header)    |
| `.collab-channel-list`| `flex: 1 1 auto`, `overflow-y: auto`        |
| `.collab-main`        | column-flex, `overflow: hidden`             |
| `.collab-chat-head`   | `flex: 0 0 auto` (sticky channel header)    |
| `.collab-messages`    | `flex: 1 1 auto`, `overflow-y: auto`        |
| `.collab-typing`      | `flex: 0 0 auto`                            |
| `.collab-composer`    | `flex: 0 0 auto` (pinned at bottom)         |

Plus `body.collab-active { overflow: hidden }` to suppress the page
scrollbar while the user is on Team Chat — the only scrollable areas
are now the channel list and the messages list.

`overscroll-behavior: contain` is set on both scroll panes so a
two-finger swipe in messages doesn't bounce the whole window on
trackpads / touch screens.

Both panes also get a subtle 8-px scrollbar treatment (light and dark)
matching the visual hierarchy of the surrounding panels.

### Responsiveness

The fix only applies at **≥901px**. Below that, the existing media
query at `max-width: 900px` already stacks the sidebar above the main
pane (`grid-template-columns: 1fr`) and lets the channel list cap at
`260px`. On mobile, the natural page-level scroll is preserved, which
is what users expect from chat apps on phones.

### Auto-scroll preserved

`loadCollabMessages(true)` continues to call
`wrap.scrollTop = wrap.scrollHeight` to jump to the newest message.
With the new `flex: 1 1 auto; min-height: 0` constraint on
`.collab-messages`, the scrollable area is now sized correctly, so
that jump lands the bottom of the list flush against the composer.

## Issue #4 — Answered: longer search bar, visible clear button

### Reported behavior

The search bar on the **Answered** page felt visually narrow even on
wide screens, and the user wanted a clear/× control inside it.

### Root cause

A previous responsive pass had set `.faq-search-wrap { width: min(760px,
100%); max-width: 760px; }`, but the inner clear-wrapper
(`.search-clear-wrap`) was `display: inline-flex` — which is
**content-width**, not fill-width. The author of the previous patch had
added an inline `style="flex:1"` on that wrapper hoping to expand it,
but the parent `.faq-search-wrap` wasn't a flex container, so `flex:1`
was silently ignored.

Result: the wrapper shrank to about the natural width of the input
element regardless of the 760px outer container, and on wide displays
the search field looked stranded with empty space on either side.

A clear "×" button already existed in markup — it just appeared near
the left side of the empty space because of the shrunken wrapper.

### Fix

- `.faq-search-wrap` is now `display: flex; align-items: center` and
  widened to `min(880px, 100%); max-width: 880px`.
- The direct-child `.search-clear-wrap` inside it is now
  `display: flex; flex: 1 1 auto; min-width: 0` so it fills the
  available width and the input properly stretches to `width: 100%`.
- The × button is nudged to `right: 14px` so it sits cleanly inside
  the pill's rounded right edge.
- The `.search-clear-wrap` rule used in the **Filters Row** of the
  All Questions / Review / Archive pages is **untouched** — the
  scoping selector is `.faq-search-wrap > .search-clear-wrap`, so
  no regression for those rows.

## Targeted audit pass — related flows

These issues weren't reported, but they share a root cause with #1 and
#2 and are fixed by the same patch as a side effect:

### 4.1  Links inside the question cell

`<a>` elements rendered from URLs in question text also bubbled up to
the `<tr onclick>` of All Questions / Review tables, so following a
link would also navigate / open the modal. The patch now stops
propagation on every `<a>` and every `.bugfix-attachment-link`,
`.rich-attachment-link`, `.rich-link-preview`, and
`.rich-image-attachment` found inside `.td-q`.

### 4.2  Answered (FAQ) cards: image chevron triggering the card collapse

The `toggleFaqCard` override (line 4441) excluded
`.bugfix-attachment-image, .bugfix-attachment-image-wrap, .faq-question,
.faq-answer-preview` from triggering the card-collapse — but did **not**
exclude `.img-collapse` (the `<details>` wrapper added later by the
image-collapse script). Clicking the chevron on a FAQ card therefore
collapsed the card.

The patch's `.img-collapse → stopPropagation` listener fixes this at the
source, so the exclusion list in `toggleFaqCard` no longer needs
updating — the click never reaches the card.

### 4.3  Accessibility polish on the expand chevron

The `<summary>` rendered with no visual affordance for hover/focus. The
patch adds:

- a soft `--surface3` hover background on the summary,
- a 2-px `--accent2` outline on `:focus-visible` (keyboard tab),
- a tiny radius/padding so the hover target reads as a button-like chip.

Native keyboard activation (Enter / Space on a focused summary) was
already working and is preserved.

### 4.4  Scrollbar polish on Team Chat panes

A consistent 8-px scrollbar treatment (light + dark modes) is applied
to `.collab-channel-list` and `.collab-messages` so the two scroll
zones read as scrollable but don't dominate the visual hierarchy.

### 4.5  Body scroll lock scoped to Team Chat only

To prevent double scrollbars on Team Chat, `body.collab-active` is
toggled by hooking `showPage()`. The class is only added while the
Team Chat tab is active — every other page keeps its natural
page-level scroll, so this does not regress the Submit / All
Questions / Review / Answered / Admin / Privacy pages.

## Known follow-ups (deferred, not in this patch)

These are not regressions but were noted during the audit; they
deserve their own ticket:

- **`localStorage` route persistence** can briefly flash the wrong
  page on login because `showPage()` runs before `currentUser` is
  fully resolved. Mitigation is already in place via
  `canAccessRoute()`, but a small loading state would feel more
  premium.
- **`MutationObserver` count** — the file now has three: image
  attachments renderer, image-collapse wrapper, and this bugfix
  sweeper. They are all idempotent and short-running, so the perf
  impact is negligible, but a future refactor could consolidate them
  into a single observer dispatching to all three handlers.
- **`<tr onclick>` pattern** — the safer long-term shape is to use
  delegated handlers on the `<tbody>` checking `event.target.closest`
  against an allow-list (or a `[data-row-clickable]` attribute on the
  row's clickable zones). That's a larger refactor; this patch leaves
  the existing inline `onclick` in place and just stops the wrong
  children from triggering it.
