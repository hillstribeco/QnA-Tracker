# Bug Fixes — 2026-05-26

Three related bugs in the comment/attachment system were reported and fixed.

## Bugs reported

1. **The × button on attachment previews didn't react.** Attached images/files could only be removed by refreshing the page.
2. **Uploaded images appeared as a long block of text** (the raw base64 data URL) instead of an inline image preview.
3. **None of the comment-form buttons reacted** — `+ Upload`, `🔗 Link`, `😊 Emoji`, `Reply`, and `Clear` — on the Follow-up, Answered (FAQ), and Review pages.

These were pre-existing bugs in the original `index_production_refactored_qa_hardened.html`. They were NOT introduced by the project restructuring.

## Root causes

### Bugs 1 & 3: a single broken event listener

A document-level `click` listener registered with **capture-phase = true** was calling `e.stopPropagation()` on every click inside `.comments-section button`, `.comments-section .communication-toolbar`, or `.attachment-preview-strip`:

```javascript
// The buggy code:
document.addEventListener('click', e => {
  if (e.target.closest?.('.comments-section button, .comments-section textarea, .comments-section input, .comments-section .communication-toolbar, .attachment-preview-strip')) e.stopPropagation();
}, true);
```

The author likely intended to prevent the FAQ card's `onclick` (which collapses the card) from firing when clicking inside the comments section. But this approach has a critical flaw:

**When `stopPropagation()` is called during the capture phase by an ancestor (document), the event never reaches the target element's own listeners — including inline `onclick="..."` attributes.**

This is verified by the DOM dispatch algorithm and reproducible in a JSDOM test (see `/home/claude/test_fix.js` if you want to run it yourself).

The result: every button inside a comment form, the communication toolbar, or an attachment preview strip became un-clickable. The state changes (like adding a file to the attachment store) still happened on other code paths (e.g. drag-drop, focus events), but the inline button handlers never fired.

The original "don't collapse the FAQ card" intent was **already** correctly handled inside `toggleFaqCard()` itself, which short-circuits when the click target is inside `.comments-section, button, textarea, input, select, a`. So the broken document-level listener was redundant *and* broken.

### Bug 2: attachment markdown was never converted back into images

When you upload an image to a comment, the existing code converts it to a base64 data URL and inserts text like this into the textarea:

```
[Attachment: screenshot.png](data:image/png;base64,iVBORw0KGgoAAAANS...)
```

When the comment is submitted, this raw text is saved to the `question_comments.text` column in Supabase. When displayed, the comment renderer does:

```javascript
<div class="comment-text" ...>${escHtml(node.text)}</div>
```

`escHtml` only HTML-escapes the text. It doesn't convert the `[Attachment: …](url)` markdown back into an `<img>` or `<a>` tag. So users saw the raw markdown — and for images, a multi-thousand-character base64 string.

(The other existing function `renderMentionedText` handles `@username` patterns but not attachments, and is only called on FAQ questions/answers, not on comment bodies.)

## Fixes applied

### Fix 1: removed the broken capture-phase listener

In `frontend/index.html`, the 3 offending lines were removed and replaced with a comment explaining why. No other code was changed for this fix.

### Fix 2: a self-contained attachment-rendering script

A new `<script>` block was added at the end of the `<body>` that:

1. Defines `window.renderAttachmentMarkdown(text)` — turns `[Attachment: name](url)` patterns into either:
   - An `<img>` tag if the URL is `data:image/*`, `blob:`, or ends in a recognized image extension.
   - An `<a download>` link otherwise.
   - **Unsafe URL schemes** (`javascript:`, `file:`, etc.) are escaped as plain text. This is a security improvement on top of the bug fix.
2. Walks all `.comment-text`, `.faq-question`, `.faq-answer-preview`, `.detail-box`, and `.collab-bubble-body` elements once each, applying the conversion idempotently via a `data-bugfix-attachments-rendered` flag.
3. Watches for new such elements via a `MutationObserver`, so newly-rendered comments are enhanced automatically.
4. Wraps the key render functions (`refreshComments`, `loadFaqComments`, `loadModalComments`, `renderFaq`, `renderReviewTable`, `renderAllQ`, `loadCollabMessages`) so enhancement happens immediately after a render.
5. Adds CSS that gives the rendered images a clean, capped display (max-height 320px, rounded border, caption underneath).
6. Adds a click-to-open-full-size behavior — clicking a rendered image opens it in a new tab at full resolution.

Because this fix is **purely additive**, removing the entire script block reverts the behavior to "raw markdown text". If anything misbehaves, you can comment out or delete this block without affecting anything else.

## Validation

Reproduced both bugs in a JSDOM test harness. Verified:

- **Before fix:** `Reply` / `Clear` / `×` button onclicks all return `false` (handlers don't fire). Confirms the bug.
- **After fix:** All three onclick handlers fire and set their flags. Confirmed the fix.
- **Image rendering:** A `data:image/png;base64,...` URL becomes an `<img src="…">` tag. A `https://…/file.pdf` URL becomes an `<a href="…">` tag. A `javascript:alert(1)` URL is escaped as plain text (not made clickable).

## What was NOT fixed (deferred)

The current implementation still stores uploaded images as base64 data URLs inside the `question_comments.text` column. For an internal tool with a small number of images this works, but it's not ideal:

- Multi-megabyte rows make queries slower over time.
- Backups grow quickly.
- Mobile network usage to load a comment thread can be heavy.

The proper long-term fix is to use **Supabase Storage** for image uploads: upload to a storage bucket, store only the storage path in the database. That requires:

1. Creating a storage bucket (e.g. `comment-attachments`).
2. Adding RLS policies on the bucket.
3. Changing the upload flow (`processAttachmentFiles` in the inline JS) to call `sb.storage.from(...).upload(...)` instead of reading the file as a base64 data URL.
4. Resolving stored paths to public URLs at render time.

This is in the `docs/MAINTENANCE.md` "Future improvements" section as **Add Supabase Storage support for question image attachments** — estimated 3-4 hours of work.

## How to roll back

If anything misbehaves after these fixes:

1. The capture listener removal: search `index.html` for `BUGFIX 2026-05-26` and the comment explaining why the listener was removed. To restore the listener, replace the comment with the original 3 lines (visible in the bug-report section of this document).
2. The attachment rendering script: delete the entire `<script>` block at the bottom of the file that contains `attachmentRenderingFix` and the related `<style>` block. The app reverts to "raw markdown" display but still works.

Both changes are isolated and reversible.
